Manual do projeto — implementação e porquês

Não é só "o que" cada função faz — é por que o projeto foi montado dessa forma, e em que ordem faz sentido construir cada peça.

Por que duas pilhas?CONCEITO

O desafio do push_swap não é "ordenar números" — isso é trivial. O desafio é ordenar usando o mínimo de movimentos possível, restrito a duas estruturas de dados (pilhas, LIFO) e um alfabeto fixo de 11 operações.

Uma única pilha não seria suficiente: com só uma pilha você só consegue rotacionar ou trocar os dois primeiros elementos, mas nunca "estacionar" um valor de lado enquanto reorganiza o resto. A segunda pilha (B) funciona como uma área de trabalho temporária — é o que permite, por exemplo, tirar os menores valores do caminho (sort_five) ou processar a lista em faixas (sort_big).

Por quê importa: entender isso primeiro evita tentar "forçar" uma solução de ordenação genérica (bubble sort, quicksort) direto na pilha A — o problema real é de movimentação entre duas pilhas, não de comparação.
Por que exatamente essas 11 operações?CONCEITO

As 11 operações não foram escolhidas ao acaso — elas cobrem o conjunto mínimo e completo de movimentos necessários para qualquer rearranjo possível entre duas pilhas:

  • Trocar os 2 primeiros (sa/sb/ss) — corrige inversões locais sem gastar uma rotação inteira.
  • Mover entre pilhas (pa/pb) — é a única forma de tirar um valor "do caminho".
  • Rotacionar pra cima (ra/rb/rr) — traz o próximo elemento ao topo sem precisar de push.
  • Rotacionar pra baixo (rra/rrb/rrr) — o inverso, útil quando o elemento alvo está perto do fim.

As versões combinadas (ss, rr, rrr) existem só por eficiência de contagem: fazer sa+sb como duas operações separadas custa 2 no benchmark; ss custa 1. Isso é decisivo pra bater as metas de menos de 700/5500 operações.

Por que sort_big usa "chunks" (Turk) em vez de um sort genérico?DECISÃO DE DESIGN

Um quicksort ou insertion sort "puro" pressupõe acesso aleatório aos elementos — você pode comparar o índice 3 com o índice 47 diretamente. Numa pilha, isso não existe: só se enxerga o topo. Qualquer comparação exige rotacionar até o elemento certo, o que custa operações.

O algoritmo Turk contorna isso dividindo o intervalo de valores (não os índices) em faixas — os chunks. Cada valor é empurrado pra B assim que sua faixa é alcançada, em ondas. Isso transforma "onde este valor deveria estar" numa decisão que só olha o topo da pilha, sem precisar de acesso aleatório.

Armadilha comum: escolher um chunk_size fixo (ex: sempre 20) em vez de calculá-lo a partir do tamanho real da pilha (size / 5) faz o algoritmo estourar a meta de operações em listas grandes.
Erros comunsARMADILHAS
free_stack sem guardar o próximo antes: se você faz free(node); node = node->next; nessa ordem, é use-after-free — o next já foi liberado quando você tenta lê-lo.
ss/rr/rrr chamando sa()/sb() por dentro: isso imprime "sa" e "sb" separadamente em vez de uma linha só de "ss" — o código precisa ser duplicado (sem print) e não reaproveitado via chamada de função.
Não validar overflow no ft_atoi: um valor como "2147483648" (INT_MAX + 1) precisa ser pego antes do cast — se você só faz atoi() puro, o comportamento é indefinido.

typedef struct s_list { int value; struct s_list *next; } t_list;
sa — troca os 2 primeiros de AGRUPO 1
void sa(t_list **stack_a) { t_list *first; t_list *second; if (!stack_a || !*stack_a || !(*stack_a)->next) return ; first = *stack_a; second = first->next; first->next = second->next; second->next = first; *stack_a = second; ft_putstr("sa\n"); }
  1. t_list *first; t_list *second; — declaro os dois ponteiros auxiliares no topo da função (Norma exige declaração antes de qualquer instrução).
  2. if (!stack_a || !*stack_a || !(*stack_a)->next) return; — três checagens em cascata: o ponteiro pro ponteiro existe? a pilha existe? ela tem um segundo elemento? Se qualquer uma falhar, não tem o que trocar.
  3. first = *stack_a; — guardo o nó do topo atual (ex: 5) antes de mexer em qualquer next. Se mexesse primeiro, perderia a referência.
  4. second = first->next; — guardo o segundo nó (ex: 3). Vai virar o novo topo.
  5. first->next = second->next; — o pulo do gato: first (5) passa a apontar pra onde second apontava, pulando o second. Isso tira o 5 do meio da corrente.
  6. second->next = first; — second (3) aponta pro first (5). A ordem virou 3 → 5 → resto.
  7. *stack_a = second; — atualizo o ponteiro da pilha: o topo agora é second. Sem essa linha, a pilha continuaria "achando" que o topo é o antigo first.
  8. ft_putstr("sa\n"); — só imprime depois que a troca já aconteceu de verdade.
sb: idêntico, trocando stack_a por stack_b em toda a função — não tem lógica nova pra aprender.

Se eu chamasse as duas, cada uma imprimiria sua própria linha — "sa\nsb\n" no lugar de "ss\n" — e isso conta como 2 operações no benchmark em vez de 1. Por isso o código é duplicado aqui dentro, sem print, com ft_putstr("ss\n") uma única vez no final.
pa — move o topo de B pra AGRUPO 2
void pa(t_list **stack_a, t_list **stack_b) { t_list *first; if (!stack_b || !*stack_b) return ; first = *stack_b; *stack_b = first->next; first->next = *stack_a; *stack_a = first; ft_putstr("pa\n"); }
  1. if (!stack_b || !*stack_b) return; — só precisa checar 1 elemento em B (diferente do swap, aqui não precisa de 2).
  2. first = *stack_b; — guardo o nó do topo de B.
  3. *stack_b = first->next; — atualizo B primeiro, antes de mexer no next do first. Na ordem errada, eu perderia a referência pro resto de B.
  4. first->next = *stack_a; — conecto o nó que saiu de B ao topo atual de A.
  5. *stack_a = first; — o nó que veio de B vira o novo topo de A.
pb: espelho exato — troca stack_a ↔ stack_b em cada linha.
ra — primeiro vira últimoGRUPO 3
void ra(t_list **stack_a) { t_list *first; t_list *last; if (!stack_a || !*stack_a || !(*stack_a)->next) return ; first = *stack_a; *stack_a = first->next; first->next = NULL; last = *stack_a; while (last->next) last = last->next; last->next = first; ft_putstr("ra\n"); }
  1. first = *stack_a; — guardo o topo (vai virar o último).
  2. *stack_a = first->next; — a pilha "esquece" o first; o segundo elemento vira o novo topo.
  3. first->next = NULL; — crítico: isolo o first. Sem isso, ele ainda apontaria pro antigo segundo elemento, e ao encaixar no final eu criaria um loop infinito na lista.
  4. last = *stack_a; while (last->next) last = last->next; — percorro a lista até achar o nó cujo next é NULL — esse é o último.
  5. last->next = first; — o último passa a apontar pro first, que agora é o novo último de verdade.
rb: idêntico, trocando stack_a por stack_b.
rra — último vira primeiroGRUPO 4
void rra(t_list **stack_a) { t_list *prev; t_list *last; if (!stack_a || !*stack_a || !(*stack_a)->next) return ; prev = NULL; last = *stack_a; while (last->next) { prev = last; last = last->next; } prev->next = NULL; last->next = *stack_a; *stack_a = last; ft_putstr("rra\n"); }
  1. prev = NULL; last = *stack_a; — dois ponteiros andando juntos: last sempre um passo à frente de prev.
  2. while (last->next) { prev = last; last = last->next; } — a cada volta, prev vira o antigo last, e last avança. Quando o loop para, last é o último nó e prev é o penúltimo — por isso precisa dos dois, não dá pra "voltar" numa lista simplesmente encadeada.
  3. prev->next = NULL; — corto a ligação do penúltimo com o último, isolando o último.
  4. last->next = *stack_a; — o último passa a apontar pro antigo topo.
  5. *stack_a = last; — o último vira o novo topo.
rrb: idêntico trocando stack_a por stack_b. rrr: mesma pegadinha da ss/rr — duplica rra + rrb sem chamá-las, print único no final.
Norma: rr, rrr e ss declaram first/second/last/prev uma vez só e reaproveitam pros dois blocos (A e B) — a Norma não deixa declarar a mesma variável duas vezes na função.
ft_atoi — converte texto em inteiroFUNC. 01
int ft_atoi(const char *str, int *error) { long num; int sign; int i; num = 0; sign = 1; i = 0; *error = 0; while (str[i] == ' ' || (str[i] >= 9 && str[i] <= 13)) i++; if (str[i] == '+' || str[i] == '-') { if (str[i] == '-') sign = -1; i++; } if (!(str[i] >= '0' && str[i] <= '9')) { *error = 1; return (0); } while (str[i] >= '0' && str[i] <= '9') { num = num * 10 + (str[i] - '0'); if (num * sign > INT_MAX || num * sign < INT_MIN) { *error = 1; return (0); } i++; } if (str[i] != '\0') { *error = 1; return (0); } return ((int)(num * sign)); }
  1. long num; — uso long, não int, de propósito: preciso comparar contra INT_MAX/INT_MIN antes de estourar, e um int já estourado vira lixo antes mesmo da comparação.
  2. *error = 0; — inicializo o output assumindo sucesso; só viro 1 nos pontos de falha.
  3. while (str[i] == ' ' || (str[i] >= 9 && str[i] <= 13)) — pula espaços e os outros espaços em branco (tab, newline etc. — códigos ASCII 9 a 13), igual o atoi da libc faz.
  4. if (str[i] == '+' || str[i] == '-') — captura um sinal opcional, guardando em sign antes de avançar i.
  5. if (!(str[i] >= '0' && str[i] <= '9')) { *error = 1; return (0); } — se depois do sinal não vier nenhum dígito, a string é inválida (ex: "--" ou "").
  6. num = num * 10 + (str[i] - '0'); — a conversão clássica dígito a dígito, deslocando uma casa decimal a cada iteração.
  7. if (num * sign > INT_MAX || num * sign < INT_MIN) — checagem de overflow feita dentro do loop, a cada dígito — não dá pra esperar o número inteiro ser montado, porque nesse ponto ele já pode ter estourado o long também em casos extremos.
  8. if (str[i] != '\0') { *error = 1; return (0); } — depois dos dígitos, se ainda sobrar algum caractere (tipo o a de "42a"), a entrada é inválida.
  9. return ((int)(num * sign)); — só faço o cast de volta pra int no final, já validado.
create_node — aloca um nóFUNC. 02
t_list *create_node(int value) { t_list *node; node = malloc(sizeof(t_list)); if (!node) return (NULL); node->value = value; node->next = NULL; return (node); }
  1. node = malloc(sizeof(t_list)); — aloca exatamente o tamanho do struct.
  2. if (!node) return (NULL); — sempre checa o retorno de malloc. Sem isso, um `malloc` que falha (memória esgotada) vira um crash silencioso mais tarde, quando você tentar usar `node->value`.
  3. node->value = value; node->next = NULL; — inicializa os dois campos. Esquecer o next = NULL é a causa nº1 de segfault em listas encadeadas — sem isso o ponteiro fica com lixo de memória.
add_to_stack — encaixa no final da listaFUNC. 03
void add_to_stack(t_list **stack, int value) { t_list *node; t_list *last; node = create_node(value); if (!node) return ; if (!*stack) { *stack = node; return ; } last = *stack; while (last->next) last = last->next; last->next = node; }
  1. node = create_node(value); — reaproveita a função anterior, não duplica a lógica de alocação.
  2. if (!*stack) { *stack = node; return; } — caso especial: lista vazia. Sem esse atalho, o while (last->next) logo abaixo quebraria tentando ler (*stack)->next com *stack igual a NULL.
  3. last = *stack; while (last->next) last = last->next; — percorre até achar o nó sem sucessor.
  4. last->next = node; — encaixa o novo nó no final, preservando a ordem de leitura do argv.
check_duplicates — dois ponteiros aninhadosFUNC. 04
int check_duplicates(t_list *stack) { t_list *current; t_list *check; current = stack; while (current) { check = current->next; while (check) { if (current->value == check->value) return (1); check = check->next; } current = current->next; } return (0); }
  1. current = stack; — ponteiro externo, percorre nó a nó.
  2. check = current->next; — ponteiro interno, sempre começa depois de current — assim cada par só é comparado uma vez (não compara um nó com ele mesmo, nem repete pares na ordem inversa).
  3. if (current->value == check->value) return (1); — no primeiro par igual encontrado, já retorna — não precisa continuar procurando.
  4. return (0); — só chega aqui se os dois loops terminarem sem achar nenhum par igual.
Custo: isso é O(n²) — para o tamanho de lista deste projeto (até algumas centenas de números) é aceitável; não vale a complexidade de um hash set só pra essa checagem.
parse_args — amarra tudoFUNC. 05
int parse_args(int argc, char **argv, t_list **stack_a) { int i; int num; int error; i = 0; error = 0; while (++i < argc) { num = ft_atoi(argv[i], &error); if (error) return (0); add_to_stack(stack_a, num); } if (check_duplicates(*stack_a)) return (0); return (1); }
  1. while (++i < argc) — pré-incremento: i começa em 0, mas o primeiro valor testado já é 1, pulando argv[0] (o nome do programa).
  2. num = ft_atoi(argv[i], &error); if (error) return (0); — converte e já sai no primeiro erro — não faz sentido continuar processando uma entrada que já sabemos ser inválida.
  3. add_to_stack(stack_a, num); — só chega aqui se a conversão foi válida.
  4. if (check_duplicates(*stack_a)) return (0); — a checagem de duplicados só faz sentido depois que a lista inteira já foi montada — não dá pra checar duplicado contra números que ainda não foram lidos.
main — orquestra tudoFUNC. 01
int main(int argc, char **argv) { t_list *stack_a; t_list *stack_b; stack_a = NULL; stack_b = NULL; if (argc < 2) return (0); if (!parse_args(argc, argv, &stack_a)) { print_error(); free_stack(&stack_a); return (1); } if (is_sorted(stack_a)) { free_stack(&stack_a); return (0); } push_swap(&stack_a, &stack_b); free_stack(&stack_a); free_stack(&stack_b); return (0); }
  1. stack_a = NULL; stack_b = NULL; — inicializa os dois ponteiros. Uma pilha em C é só um ponteiro pro primeiro nó — NULL significa "pilha vazia".
  2. if (argc < 2) return (0); — sem argumentos, não há o que ordenar. Retorna sucesso (não é erro rodar ./push_swap sem nada).
  3. if (!parse_args(...)) { print_error(); free_stack(&stack_a); return (1); } — se o parsing falhar (overflow, caractere inválido, duplicado), imprime o erro no stderr, libera o que já tinha sido alocado até então, e sai com código 1.
  4. if (is_sorted(stack_a)) { free_stack(&stack_a); return (0); } — se a entrada já vier ordenada, sai em silêncio — nenhuma operação precisa ser impressa.
  5. push_swap(&stack_a, &stack_b); — só chega aqui se sobrou trabalho de verdade pra fazer. Cada operação chamada dentro dela já imprime sua própria linha.
  6. free_stack(&stack_a); free_stack(&stack_b); — libera as duas pilhas antes de sair, mesmo que stack_b tenha ficado vazia no meio do processo (a função já trata esse caso).
Por que checar is_sorted antes de chamar push_swap: se você deixasse essa checagem só dentro do dispatcher (push_swap), teria que duplicá-la em cada algoritmo (sort_three, sort_five, sort_big). Fazer uma vez só aqui, no main, evita isso.
is_sorted / stack_size / get_indexTRAVESSIAS SIMPLES
int is_sorted(t_list *stack) { while (stack && stack->next) { if (stack->value > stack->next->value) return (0); stack = stack->next; } return (1); } int stack_size(t_list *stack) { int size; size = 0; while (stack) { size++; stack = stack->next; } return (size); } int get_index(t_list *stack, int value) { int i; i = 0; while (stack) { if (stack->value == value) return (i); i++; stack = stack->next; } return (-1); }
  1. while (stack && stack->next) — em is_sorted, o && importa: sem checar stack primeiro, uma lista vazia (NULL) causaria segfault ao tentar ler stack->next.
  2. if (stack->value > stack->next->value) return (0); — compara cada par de vizinhos; a primeira inversão encontrada já é suficiente pra saber que não está ordenado.
  3. stack_size e get_index seguem o mesmo molde: reatribuir stack = stack->next a cada volta é o padrão de travessia de lista encadeada em C — sem isso, o loop nunca termina.
  4. get_index retorna -1 se não achar — esse valor sentinela é o que sort_five usa pra saber que algo deu errado, já que -1 nunca é um índice válido.
find_min / find_maxMESMO MOLDE, SINAL TROCADO

  1. min = stack->value; — começa assumindo que o primeiro valor é o menor, em vez de usar um sentinela tipo INT_MAX — mais simples e não depende de nenhum valor mágico.
  2. if (stack->value < min) min = stack->value; — atualiza sempre que encontra algo menor.

free_stack — libera sem use-after-freeGERENCIAMENTO DE MEMÓRIA
void free_stack(t_list **stack) { t_list *tmp; if (!stack || !*stack) return ; while (*stack) { tmp = (*stack)->next; free(*stack); *stack = tmp; } }
  1. tmp = (*stack)->next; — guarda o próximo nó antes de liberar o atual. Essa ordem não é opcional.
  2. free(*stack); — libera o nó atual. Depois dessa linha, ler (*stack)->next seria use-after-free — por isso o tmp já foi salvo na linha anterior.
  3. *stack = tmp; — avança o topo pro nó seguinte, e o loop repete até *stack virar NULL.
Ordem errada comum: free(*stack); *stack = (*stack)->next; — nessa ordem, a segunda linha lê a memória que acabou de ser liberada na primeira. É undefined behavior, e às vezes "funciona por sorte" em debug, mas quebra em produção.

push_swap — o dispatcherFUNC. 00
void push_swap(t_list **stack_a, t_list **stack_b) { int size; size = stack_size(*stack_a); if (size <= 1 || is_sorted(*stack_a)) return ; if (size <= 3) sort_three(stack_a); else if (size <= 5) sort_five(stack_a, stack_b); else sort_big(stack_a, stack_b); }
  1. if (size <= 1 || is_sorted(*stack_a)) return; — segunda camada de proteção (a primeira já está no main) — útil se essa função for chamada de outro lugar (testes, por exemplo).
  2. Os três if/else if/else escolhem o algoritmo certo pelo tamanho — cada limite (3 e 5) existe porque acima deles o número de casos possíveis explode, e um algoritmo de força bruta (comparar tudo com tudo) deixaria de caber no orçamento de operações.
sort_three — 6 casos fixosFUNC. 01
void sort_three(t_list **stack_a) { int a; int b; int c; a = (*stack_a)->value; b = (*stack_a)->next->value; c = (*stack_a)->next->next->value; if (a > b && b < c && a < c) sa(stack_a); else if (a > b && b < c && a > c) ra(stack_a); else if (a < b && b > c && a > c) rra(stack_a); else if (a < b && b > c && a < c) { sa(stack_a); ra(stack_a); } else if (a > b && b > c) { sa(stack_a); rra(stack_a); } }
  1. a = (*stack_a)->value; ... — lê os 3 valores direto, sem rotacionar nada ainda. Com só 3 elementos, dá pra decidir tudo de uma vez só olhando os valores.
  2. Cada if testa uma combinação de comparações (a>b, b<c, a<c) que identifica exatamente 1 das 6 permutações possíveis de 3 números distintos.
  3. a > b && b < c && a < c → ex: [2,1,3] — só uma troca resolve: sa.
  4. a > b && b < c && a > c → ex: [3,1,2] — o menor já está no meio certo, só precisa girar: ra.
  5. a < b && b > c && a > c → ex: [2,3,1] — o maior está no meio, girar pro outro lado é mais direto: rra.
  6. Os dois últimos casos precisam de 2 operações (sa+ra ou sa+rra) porque nenhuma rotação ou troca única resolve sozinha — é sempre o mínimo possível pra cada permutação.
  7. Se nenhum if disparar, é porque a < b < c — já está ordenado, e a função não faz nada.
sort_five — usa B como área de esperaFUNC. 02
void sort_five(t_list **stack_a, t_list **stack_b) { int min; int index; int size; while (stack_size(*stack_a) > 3) { size = stack_size(*stack_a); min = find_min(*stack_a); index = get_index(*stack_a, min); if (index <= size / 2) { while (index-- > 0) ra(stack_a); } else { while (index++ < size) rra(stack_a); } pb(stack_a, stack_b); } sort_three(stack_a); while (*stack_b) pa(stack_a, stack_b); }
  1. while (stack_size(*stack_a) > 3) — repete até sobrarem só 3 elementos em A, que é quando sort_three pode assumir o resto.
  2. min = find_min(*stack_a); index = get_index(*stack_a, min); — acha o menor valor de A e sua posição atual.
  3. if (index <= size / 2) { while (index-- > 0) ra(stack_a); } — se o menor está na metade de cima da pilha, gira pra cima (ra) o número exato de vezes até ele chegar ao topo.
  4. else { while (index++ < size) rra(stack_a); } — se está mais perto do final, gira pra baixo (rra) em vez de dar a volta inteira — economiza operações.
  5. pb(stack_a, stack_b); — com o menor já no topo, empurra pra B.
  6. sort_three(stack_a); — depois do loop, sobraram exatamente 3 em A — resolve com a função que já construímos.
  7. while (*stack_b) pa(stack_a, stack_b); — devolve os elementos de B pra A um a um. Como eles foram empurrados em ordem crescente (do menor pro maior), devolver na ordem de pilha (LIFO) já reconstitui a ordem certa.
Por que escolher ra ou rra: girar pra cima ou pra baixo custa o mesmo tipo de operação, mas girar na direção mais curta (metade mais próxima) sempre gasta menos passos — é essa escolha que ajuda a bater a meta de operações no benchmark.
sort_big — o algoritmo Turk (chunks)FUNC. 03
void sort_big(t_list **stack_a, t_list **stack_b) { int size; int chunk; int i; int max; size = stack_size(*stack_a); chunk = size / 5; i = -1; max = size - 3; while (++i < max) { if ((*stack_a)->value <= i + chunk) pb(stack_a, stack_b); else if ((*stack_a)->value <= i + chunk * 2) { pb(stack_a, stack_b); rb(stack_b); } else ra(stack_a); } sort_three(stack_a); while (*stack_b) { pa(stack_a, stack_b); if (stack_size(*stack_a) > 1 && (*stack_a)->value > (*stack_a)->next->value) sa(stack_a); } while ((*stack_a)->value != find_min(*stack_a)) ra(stack_a); }
  1. chunk = size / 5; — define o tamanho de cada faixa de valores. Dividir por 5 é empírico: testes mostram que esse número de faixas equilibra bem o número de rotações contra o número de pushes, pra listas de 100+ elementos.
  2. max = size - 3; — para o loop principal quando sobrarem só 3 elementos em A — igual no sort_five, é o ponto onde sort_three assume.
  3. if ((*stack_a)->value <= i + chunk) pb(stack_a, stack_b); — se o valor do topo pertence à faixa atual, empurra direto pra B.
  4. else if (... <= i + chunk * 2) { pb(...); rb(stack_b); } — se pertence à próxima faixa, empurra e já gira B uma vez — isso mantém B parcialmente organizado à medida que cresce, em vez de deixar tudo bagunçado pra reorganizar depois.
  5. else ra(stack_a); — se o valor não pertence a nenhuma das duas faixas próximas, gira A pra trazer outro candidato ao topo, sem gastar um push "errado".
  6. sort_three(stack_a); — organiza os 3 que sobraram em A.
  7. while (*stack_b) { pa(...); if (...) sa(stack_a); } — devolve tudo de B pra A; a checagem com sa corrige a ordem local caso o valor recém-chegado esteja fora de posição em relação ao que já está em A.
  8. while ((*stack_a)->value != find_min(*stack_a)) ra(stack_a); — rotação final: garante que o menor valor de toda a pilha fique no topo, alinhando A de vez.
Por que não usar chunk fixo: calcular chunk = size / 5 a partir do tamanho real é o que faz o algoritmo escalar de 100 pra 500 números sem estourar a meta — um número fixo (tipo sempre 20) funciona bem pra um tamanho e mal pra outro.

BUFFER_SIZE: o mesmo código, tamanhos diferentes

A avaliação (peer e Moulinette) recompila com valores de BUFFER_SIZE diferentes. Se o algoritmo depender de um tamanho específico, ele quebra.

cc -Wall -Wextra -Werror -D BUFFER_SIZE=42 get_next_line.c get_next_line_utils.c

BUFFER_SIZE = 1

read() traz 1 byte por vez. O loop de leitura roda muitas vezes pra montar uma única linha — lento, mas tem que funcionar: é o teste mais comum pra pegar bugs de concatenação.

BUFFER_SIZE = 42 (exemplo do enunciado)

Tamanho "normal", usado no exemplo de compilação do subject. Bom valor padrão pra desenvolver e debugar visualmente.

BUFFER_SIZE = 9999

Maior que muitas linhas de um arquivo de teste típico — uma única leitura pode trazer várias linhas de uma vez. O stash precisa guardar o excesso corretamente pra próxima chamada.

BUFFER_SIZE = 10000000

Testa se o malloc/read do buffer aguenta um tamanho gigante sem estourar memória ou travar — mesmo que o arquivo lido seja pequeno.

sem -D BUFFER_SIZE

O projeto precisa compilar mesmo sem a flag — por isso o enunciado permite (e espera) um valor default definido no próprio código com #ifndef.

#ifndef BUFFER_SIZE
# define BUFFER_SIZE 42
#endif
Armadilha clássica: usar strlen(buffer) depois do read() em vez do valor retornado por read(). Se o buffer tinha lixo de uma leitura anterior (comum com BUFFER_SIZE pequeno), strlen conta caracteres que não fazem parte da leitura atual.
lseek() é proibido: qualquer solução que "espie" ou "volte" no arquivo usando lseek não é válida — toda a lógica de posição tem que vir só do que já foi lido e guardado em stash.
bônus

Múltiplos file descriptors, uma variável estática

A parte mais interessante do bônus não é ler vários fds — é fazer isso sem perder o estado de nenhum, usando só uma variável static no total.

a ideia centralDESIGN

Um static char *stash sozinho só lembra de um fd por vez — se você chamar get_next_line em fd 3 e depois em fd 4, a leitura de 4 sobrescreveria o que estava guardado pra 3.

A solução mais comum: trocar o static por um array de ponteiros, indexado pelo próprio fd. Isso ainda conta como uma única variável estática (é uma declaração só), mas guarda o estado de cada fd numa posição separada.

static char *stash[OPEN_MAX]; ... stash[fd] = join_stash(stash[fd], buffer);
Por que isso satisfaz a regra: o enunciado pede "apenas uma variável estática" — não "apenas um valor". Um array é uma variável só, mesmo guardando N estados diferentes.
Cuidado: um array de tamanho fixo limita quantos fds diferentes podem ser acompanhados ao mesmo tempo. Uma alternativa mais robusta (fora do escopo mínimo) é uma lista encadeada de (fd, stash), crescendo sob demanda.

  1. len1 = gnl_strlen(s1); — se s1 for NULL, gnl_strlen já devolve 0 (primeira chamada de get_next_line).
  2. res = malloc(len1 + n + 1); — espaço pro conteúdo antigo + o pedaço novo (n bytes lidos) + o '\0' final.
  3. Primeiro while — copia o que já tinha em s1 pro início de res.
  4. Segundo while — copia os n bytes recém-lidos do buffer, logo em seguida.
  5. res[len1 + n] = '\0'; — fecha a string; sem isso, qualquer função de string subsequente leria lixo.
  6. free(s1); — libera a versão antiga do stash. Fundamental pra não vazar memória a cada chamada.
Cuidado: se malloc falhar aqui, s1 nunca é liberado — num projeto real vale tratar esse caso também, mas o enunciado tolera comportamento indefinido só em caso de erro real de sistema.
extract_line & get_remaining — separa a linha do restoHELPERS
char *extract_line(char *stash) { int i; char *line; i = 0; while (stash[i] && stash[i] != '\n') i++; if (stash[i] == '\n') i++; line = malloc(i + 1); if (!line) return (NULL); i = 0; while (stash[i] && stash[i] != '\n') { line[i] = stash[i]; i++; } if (stash[i] == '\n') line[i++] = '\n'; line[i] = '\0'; return (line); } char *get_remaining(char *stash) { int i; int j; char *rest; i = 0; while (stash[i] && stash[i] != '\n') i++; if (stash[i] == '\n') i++; if (!stash[i]) return (free(stash), NULL); rest = malloc(gnl_strlen(stash) - i + 1); if (!rest) return (free(stash), NULL); j = 0; while (stash[i]) rest[j++] = stash[i++]; rest[j] = '\0'; free(stash); return (rest); }
  1. while (stash[i] && stash[i] != '\n') i++; — em ambas as funções, o primeiro passo é achar o '\n' (ou o fim da string).
  2. if (stash[i] == '\n') i++; — inclui o '\n' na contagem, porque a linha devolvida precisa terminar com ele (quando existir).
  3. Em extract_line: aloca exatamente i bytes (o tamanho já calculado) e copia até ali — essa é a linha que get_next_line vai retornar.
  4. Em get_remaining: se depois do '\n' não sobra nada (!stash[i]), libera o stash inteiro e retorna NULL — não há mais nada guardado pra próxima chamada.
  5. Caso contrário, aloca só o restante (gnl_strlen(stash) - i bytes) e copia — vira o novo static na próxima chamada.
  6. free(stash); no fim de get_remaining — o stash antigo (já dividido em linha + resto) não é mais necessário.
Por que duas funções e não uma: extract_line não mexe em memória (só lê stash), get_remaining é quem decide o destino do stash (incluindo liberá-lo) — separar deixa claro quem é dono de qual alocação.
get_next_line — a função inteiraget_next_line.c
char *get_next_line(int fd) { static char *stash; char buffer[BUFFER_SIZE + 1]; int bytes; char *line; if (fd < 0 || BUFFER_SIZE <= 0) return (NULL); bytes = 1; while (!has_newline(stash) && bytes > 0) { bytes = read(fd, buffer, BUFFER_SIZE); if (bytes == -1) { free(stash); stash = NULL; return (NULL); } buffer[bytes] = '\0'; if (bytes > 0) stash = gnl_strjoin(stash, buffer, bytes); } if (!stash || !stash[0]) return (NULL); line = extract_line(stash); stash = get_remaining(stash); return (line); }
  1. static char *stash; — inicializada como NULL automaticamente (regra da linguagem C pra static). É o único estado que sobrevive entre chamadas.
  2. char buffer[BUFFER_SIZE + 1]; — +1 de propósito, pra sempre sobrar espaço pro '\0' depois do read().
  3. if (fd < 0 || BUFFER_SIZE <= 0) return (NULL); — validação de entrada, antes de tentar ler qualquer coisa.
  4. bytes = 1; — só pra entrar no while pela primeira vez (qualquer valor > 0 serve).
  5. while (!has_newline(stash) && bytes > 0) — continua lendo enquanto não achou '\n' E o arquivo ainda não acabou.
  6. bytes = read(fd, buffer, BUFFER_SIZE); — lê no máximo BUFFER_SIZE bytes reais do fd.
  7. if (bytes == -1) { free(stash); stash = NULL; return (NULL); } — erro de leitura: limpa tudo e desiste, sem deixar lixo pra próxima chamada.
  8. buffer[bytes] = '\0'; — fecha a string usando exatamente o que foi lido (nunca strlen(buffer), que leria lixo de uma leitura anterior).
  9. if (bytes > 0) stash = gnl_strjoin(stash, buffer, bytes); — só concatena se realmente leu algo (bytes==0 significa EOF, nada a colar).
  10. if (!stash || !stash[0]) return (NULL); — saiu do loop sem ter absolutamente nada: arquivo vazio ou já tinha terminado antes desta chamada.
  11. line = extract_line(stash); — separa a linha completa (até e incluindo o '\n', ou até o fim se não houver).
  12. stash = get_remaining(stash); — o que sobrar vira o novo estado — e a função antiga já libera a versão velha internamente.
  13. return (line); — devolve a linha pronta pro chamador.
Por que bytes > 0 no while e não só bytes != -1: quando read() retorna 0 (EOF), não faz sentido continuar chamando read() de novo — ele sempre vai devolver 0. O loop precisa parar nesse caso, mesmo sem ter achado '\n'.

lstdelone & lstclearPARTE 3
